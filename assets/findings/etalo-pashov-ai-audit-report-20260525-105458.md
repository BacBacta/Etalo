## Findings

[90] **1. Sanctioning a seller permanently locks all their in-flight escrow funds**

`EtaloEscrow._releaseItemFully` ¬∑ Confidence: 90

**Description**
`_releaseItemFully` calls `reputation.recordCompletedOrder(seller, ...)` in the Interactions phase, which reverts with `"Seller not active"` whenever the seller's `SellerStatus != Active`. Once the owner calls `applySanction()` on a seller with any in-flight shipped/arrived/delivered item, every release path (`confirmItemDelivery`, `confirmGroupDelivery`, `triggerAutoReleaseForItem`) reverts forever. `triggerAutoRefundIfInactive` only works in `Funded` state, and `forceRefund` requires `address(dispute) == address(0)` + 90-day inactivity + a registered legal hold ‚Äî none recoverable by the buyer. Funds are permanently locked for innocent buyers.

**Fix** _(implemented via ADR-054)_

```diff
  function recordCompletedOrder(address seller, uint256 orderId, uint256 amount) external onlyAuthorized {
      SellerReputation storage rep = _reputations[seller];
-     require(rep.status == SellerStatus.Active, "Seller not active");
+     if (rep.status != SellerStatus.Active) {
+         emit OrderRecorded(seller, orderId, amount);
+         return;
+     }
      ...
  }
```

[85] **2. Auto-refunded orders never release the seller's weekly volume budget**

`EtaloEscrow.triggerAutoRefundIfInactive` ¬∑ Confidence: 85

**Description**
`fundOrder` increments `sellerWeeklyVolume[seller] += order.totalAmount`, but `triggerAutoRefundIfInactive`, `forceRefund`, and `resolveItemDispute` (refund path) never decrement it. An unprivileged buyer can fund up to `MAX_SELLER_WEEKLY_VOLUME = 5,000 USDT` worth of orders against a target seller, wait 7 days, refund permissionlessly, recover their funds, and the seller has zero weekly volume remaining for the rest of the rolling week ‚Äî effectively a 7-day commerce DoS at zero cost to the attacker. Repeatable indefinitely by rotating attacker addresses.

**Fix** _(implemented via ADR-054)_

```diff
+ function _releaseSellerWeeklyVolume(address seller, uint256 amount) internal {
+     uint256 current = sellerWeeklyVolume[seller];
+     sellerWeeklyVolume[seller] = current > amount ? current - amount : 0;
+ }
```
Called from all three refund paths (`triggerAutoRefundIfInactive`, `resolveItemDispute` refund branch, `forceRefund`).

**Description**

[85] **4. Dust allocation in `createOrderWithItems` can mint `itemCommission > itemPrice` and brick release**

`EtaloEscrow.createOrderWithItems` ¬∑ Confidence: 85

**Description**
The last-item dust absorption (`itemCommission = totalCommission - sumAssigned`) is unbounded. With many small items where `itemPrice[i] * totalCommission / totalAmount` truncates to zero, the entire `totalCommission` accumulates onto the last item. Construct: 49 items at 55 wei + 1 item at 1 wei (totalAmount = 2,696 wei, totalCommission = 48 wei intra) yields `lastItemCommission = 48` on a `lastItemPrice = 1`. Every release path computes `itemNet = itemPrice - itemCommission` ‚Üí 0.8 underflow revert. The item is permanently stuck in escrow.

**Fix** _(implemented via ADR-054)_

```diff
              } else {
                  itemCommission = totalCommission - sumAssigned;
+                 if (itemCommission > itemPrices[i]) {
+                     itemCommission = itemPrices[i];
+                 }
              }
```

[80] **5. Zero-quorum N3 vote defaults to buyer win**

`EtaloVoting.finalizeVote` ¬∑ Confidence: 80

**Description**
`bool buyerWon = v.forBuyer >= v.forSeller` with both counters at 0 evaluates to `true`. After the 14-day voting period, `finalizeVote` is permissionless; if no mediator voted (apathy, weekend, holiday ‚Äî V1 launches with a small approved-mediator list per ADR-020), the dispute resolves as buyer-win with full `remainingInEscrow` refunded to the buyer per `resolveFromVote`. A malicious buyer can intentionally escalate during low-participation windows to extract refunds without any community judgment.

**Fix** _(implemented via ADR-054)_

```diff
+ require(v.forBuyer + v.forSeller > 0, "No quorum");
  bool buyerWon = v.forBuyer >= v.forSeller;
```
Paired with `EtaloDispute.adminForceCloseN3IfNoQuorum(disputeId)` (owner-only escape hatch) so a zero-quorum dispute never stays stuck at LEVEL_N3.

[80] **6. `forceRefund` orphans active dispute records and stake freezes**

`EtaloEscrow.forceRefund` ¬∑ Confidence: 80

**Description**
ADR-023 gates `forceRefund` on `address(dispute) == address(0)`, which forces the owner to call `setDisputeContract(address(0))` before refunding. Any open dispute on the order at that moment is never resolved through `_applyResolution`, so (a) `EtaloDispute._activeDisputesBySeller[seller]` never decrements (permanent `hasActiveDispute = true`), (b) `EtaloStake._freezeCount[seller]` stays >0 (`resumeWithdrawal` never fires), and (c) the seller's withdrawal cooldown is frozen forever.

**Fix** _(deferred V1.1 ‚Äî admin-recoverable, see `docs/audit/PASHOV_FINDINGS_J12.md`)_
Add an admin pre-step `adminForceCloseDispute(disputeId)` that calls `stake.resumeWithdrawal` and clears `_activeDisputesBySeller` before unsetting the dispute contract.

[80] **7. N3 escalation deadlocks when the sole approved mediator served N2**

`EtaloDispute.escalateToVoting` ¬∑ Confidence: 80

**Description**
The voter list excludes `d.n2Mediator`. With a small approved-mediator set (V1 launches with a handful per ADR-020), if the assigned N2 mediator was the only registered mediator at escalation time, `eligibleCount = 0` and the call reverts with `"No voters available"`. The dispute is stuck at `LEVEL_N2` with `block.timestamp >= d.n2Deadline`, the seller's stake stays frozen, and the buyer cannot refund.

**Fix** _(deferred V1.1 ‚Äî admin-recoverable by approving another mediator, see `docs/audit/PASHOV_FINDINGS_J12.md`)_

[80] **8. Sybil seller TVL monopolization freezes the global escrow for 7 days at attacker capital cost**

`EtaloEscrow.fundOrder` ¬∑ Confidence: 80

**Description**
`createOrderWithItems` has no seller-side consent ‚Äî any buyer can create + fund an order naming any seller address. With `MAX_TVL_USDT = 50,000` and `MAX_SELLER_WEEKLY_VOLUME = 5,000`, an attacker controlling 10 Sybil seller addresses can fund 10 √ó 5,000 USDT of orders to themselves, monopolize the global TVL cap, and prevent any other order on the platform from being funded. Permissionless `triggerAutoRefundIfInactive` releases TVL after 7 days, but the attacker can re-fill immediately by rotating Sybil sellers across a fresh weekly window. Capital lock-up: 50K USDT √ó 7 days √ó ~5% APY ‚âà $48 ‚Äî cheap relative to a market-wide commerce DoS.

**Fix** _(deferred V1.1 ‚Äî see `docs/audit/PASHOV_FINDINGS_J12.md`)_
Add per-buyer rolling cap (`MAX_BUYER_ACTIVE_USDT = 5,000`) OR shorten the inactivity window for unfunded-shipment orders.

## Leads

_Vulnerability trails with concrete code smells where the full exploit path could not be completed in one analysis pass. These are not false positives ‚Äî they are high-signal leads for manual review. Not scored._

- **Cross-border 20% release accepts any non-zero `bytes32` as shipping proof** ‚Äî `EtaloEscrow.shipItemsGrouped` ‚Äî Code smells: `proofHash != bytes32(0)` is the only validation, no on-chain registry of valid courier hashes, immediate 20% net transfer triggered. Dormant in V1 (cross-border requires seller stake, deferred per ADR-041) but live in deployed bytecode.

- **Mediator revocation race after `assignN2Mediator`** ‚Äî `EtaloDispute.approveMediator` ‚Äî Code smells: revoking an already-assigned N2 mediator does NOT clear `d.n2Mediator`. The revoked mediator can still call `resolveN2Mediation`. Unverified: list-index re-use after pop could let a different mediator satisfy `onlyAssignedMediator`.

- **Snapshot of voters at escalation time, not at vote time** ‚Äî `EtaloDispute.escalateToVoting` ‚Äî Code smells: voter list captured into `EtaloVoting._eligibility` at escalation time. Mediators added after escalation cannot vote on the live dispute; mediators revoked after escalation can still vote. 14-day drift window.

- **N1 amicable resolution can be triggered with stale buyer proposal** ‚Äî `EtaloDispute.resolveN1Amicable` ‚Äî Code smells: `_applyResolution(disputeId, refundAmount, 0)` uses the LATEST `refundAmount` from `msg.sender`, not the snapshotted matching value.

- **`isCrossBorder` is buyer-controlled and grants 2.7% commission + stake exposure to the seller** ‚Äî `EtaloEscrow.createOrderWithItems` ‚Äî Code smells: `bool isCrossBorder` from `msg.sender` with no validation against the seller's registered country.

- **No fee-on-transfer / rebasing-token guard on `usdt`** ‚Äî `EtaloEscrow.fundOrder` ‚Äî Code smells: trust `transferFrom` requested amount instead of balance-delta. Mainnet USDT is currently fee-free, but no defense if that changes.

- **`markGroupArrived` accepts seller as arrival prover** ‚Äî `EtaloEscrow.markGroupArrived` ‚Äî Code smells: `msg.sender == order.buyer || msg.sender == order.seller`. Malicious seller can self-prove arrival, kick off the `MAJORITY_RELEASE_DELAY`, then extract 90% before the parcel actually arrives.

- **`EtaloVoting.setDisputeContract` race against `finalizeVote`** ‚Äî `EtaloVoting.finalizeVote` ‚Äî Code smells: `v.finalized = true` set BEFORE the `if (disputeContract != address(0))` callback guard.

- **`EtaloEscrow._calculateCommission` rounding-to-zero enables Sybil grinding** ‚Äî `EtaloEscrow._calculateCommission` ‚Äî Code smells: truncates to 0 for `amount < 55 wei` (intra). Grinding to `TOP_SELLER_MIN_ORDERS = 50` time-gated by 60-day seniority ‚Äî bounded but cheap.
