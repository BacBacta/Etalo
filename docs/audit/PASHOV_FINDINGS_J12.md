## Findings deferred V1.1

### #3 — `EtaloStake.slashStake` can auto-UPGRADE tier when over-collateralized

**Status:** 🔧 deferred-fix · cross-border dormant in V1
**Confidence:** 85
**File:** `packages/contracts/contracts/EtaloStake.sol` line 337

**Why deferred:** Cross-border orders are deferred V2 per ADR-041,
and `slashStake` is only invoked from `EtaloDispute._applyResolution`
when a mediator awards a non-zero `slashAmount` on a cross-border
dispute. In V1 no seller will have a stake (stake gate runs only on
`isCrossBorder=true` orders), so the upgrade path is unreachable.

**V1.1 fix:**
```diff
- EtaloTypes.StakeTier newTier = _supportedTier(_stakes[seller]);
+ EtaloTypes.StakeTier supported = _supportedTier(_stakes[seller]);
+ EtaloTypes.StakeTier newTier =
+     uint8(supported) < uint8(oldTier) ? supported : oldTier;
```

### #6 — `forceRefund` orphans active disputes when admin unsets dispute contract

**Status:** 🛡️ ops-mitigation · 🔧 deferred-fix V1.1
**Confidence:** 80
**File:** `packages/contracts/contracts/EtaloEscrow.sol` line 678

**V1 ops mitigation (PRE_MAINNET_OPS.md):** before calling
`setDisputeContract(address(0))` in preparation for a `forceRefund`,
the on-call operator must:
1. Run `EtaloDispute.hasActiveDispute(seller)` for the seller(s) of
   the affected order(s).
2. If any active dispute exists, resolve it via `resolveN2Mediation`
   or `resolveN1Amicable` (with both parties signing off) BEFORE
   touching the dispute contract address. Document the resolution
   in the legal-hold reason.
3. Only then proceed with the `forceRefund` flow.

**V1.1 fix:** new admin endpoint
`adminForceCloseDispute(disputeId)` that calls
`stake.resumeWithdrawal` and clears `_activeDisputesBySeller` so the
forceRefund flow becomes safe.

### #7 — N3 escalation deadlocks when the sole approved mediator served N2

**Status:** 🛡️ ops-mitigation · 🔧 deferred-fix V1.1
**Confidence:** 80
**File:** `packages/contracts/contracts/EtaloDispute.sol` line 242

**V1 ops mitigation:** keep ≥3 approved mediators on the
`_mediatorsList` at all times. The launch checklist in
`PRE_MAINNET_OPS.md` enforces a minimum mediator count guard; if any
revocation would drop the list below 3, the operation requires a
co-signed approval.

**V1.1 fix:** in `escalateToVoting`, fall back to the full mediator
set when `eligibleCount == 0` after excluding the N2 mediator —
better one biased ballot than a permanent deadlock.

```diff
- require(eligibleCount > 0, "No voters available");
+ if (eligibleCount == 0) {
+     excludeN2 = false;
+     eligibleCount = count;
+ }
+ require(eligibleCount > 0, "No mediators registered");
```

### #8 — Sybil seller TVL monopolization (50K USDT freeze for 7 days)

**Status:** 🔧 deferred-fix V1.1
**Confidence:** 80
**File:** `packages/contracts/contracts/EtaloEscrow.sol` line 283

**V1 mitigation accepted:** the attack costs ~$48 worth of locked
capital opportunity for 7 days, and `MAX_TVL_USDT = 50,000` is a
deliberately conservative ceiling for the V1 big-bang launch
(ADR-026 + ADR-041). Hitting the global cap is a known operational
constraint; in the first weeks of V1 we expect ops to manually
expand the cap via constructor redeploy or `MAX_TVL_USDT` upgrade if
legitimate volume hits the ceiling. The Sybil-DoS variant is
detectable on-chain (single buyer, many Sybil sellers) — the indexer
J5 backend has a heuristic flag in `app/services/anomaly_detector.py`
that fires on this pattern.

**V1.1 fix options:**
- `mapping(address => uint256) buyerActiveEscrow` with
  `MAX_BUYER_ACTIVE_USDT = 5,000` cap per buyer
- Shorten `AUTO_REFUND_INACTIVE_INTRA` from 7 days → 24 hours for
  unfunded-shipment orders (compresses DoS window)
- Require seller-side `acceptOrder()` tx between `createOrderWithItems`
  and `fundOrder` (changes UX — needs ADR)

## Leads — manual review

### Cross-border 20% release accepts any non-zero `bytes32` proof

**Status:** 🔍 manual-review · cross-border dormant V1
**File:** `EtaloEscrow.shipItemsGrouped` lines 339-425

When V2 reactivates cross-border, integrate an off-chain courier API
validator on the proof hash, OR keep the 20% release behind a
feature-flag until courier integration ships.

---

### Mediator revocation race after `assignN2Mediator`

**Status:** 🔍 manual-review · 🛡️ V1 ops
**File:** `EtaloDispute.approveMediator` line 122

V1 ops: never revoke a mediator while they have an active N2
assignment. Add the check to the launch runbook.

---

### Voter set snapshot at escalation time

**Status:** 🔍 manual-review · V1 acceptable risk
**File:** `EtaloDispute.escalateToVoting` line 220

14-day voting window — mediator changes mid-vote could shape
majorities. Acceptable for V1 given the small mediator set (≤10)
and tight ops control over registry mutations.

---

### N1 amicable proposal staleness

**Status:** 🔍 manual-review
**File:** `EtaloDispute.resolveN1Amicable` line 266

Front-run scenario: buyer-then-seller-then-buyer race could lock in
a different `refundAmount` than what the seller acknowledged.
Snapshot the matched value at proposal time rather than passing
`msg.sender`'s latest value into `_applyResolution`.

---

### Buyer-controlled `isCrossBorder`

**Status:** 🔍 manual-review · cross-border dormant V1
**File:** `EtaloEscrow.createOrderWithItems` line 199

When V2 reactivates cross-border, validate the buyer-passed flag
against the seller's registered country (now stored in
`seller_profile.country` per ADR-044 / ADR-045).

---

### Fee-on-transfer / rebasing token guard on USDT

**Status:** 🔍 manual-review · USDT is fee-free on Celo mainnet
**File:** `EtaloEscrow.fundOrder` line 306

If USDT (Tether) ever migrates to a fee-on-transfer model, the
fundOrder accounting breaks. Mitigation: switch to balance-delta
accounting (`balanceAfter - balanceBefore == expected`) in V1.1.

---

### `markGroupArrived` accepts seller as arrival prover

**Status:** 🔍 manual-review · cross-border dormant V1
**File:** `EtaloEscrow.markGroupArrived` line 428

When V2 reactivates cross-border, restrict arrival proof to
buyer-only OR require dual buyer+courier signatures.

---

### `EtaloVoting.setDisputeContract` race against `finalizeVote`

**Status:** 🔍 manual-review · 🛡️ V1 ops
**File:** `EtaloVoting.finalizeVote` line 106

If `disputeContract` is `address(0)` at finalize time, the vote
silently finalizes with no callback. V1 ops: only the deployer EOA
can call `setDisputeContract`, and the runbook forbids changing it
during an active vote (block.timestamp < deadline).

---

### Sybil grinding via rounding-to-zero commission

**Status:** 🔍 manual-review · time-gated by 60d seniority
**File:** `EtaloEscrow._calculateCommission` line 973

Grinding to `TOP_SELLER_MIN_ORDERS = 50` with sub-55-wei orders that
pay 0 commission is bounded by `TIER_2_MIN_SENIORITY = 60 days`
plus `AUTO_RELEASE_INTRA = 3 days` per order. V1 acceptable.
